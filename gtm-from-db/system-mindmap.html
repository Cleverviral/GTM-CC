1. Neon DB — 5 Tables (Single Source of Truth)

Everything lives here. Clients from Tally. Leads from data team (one row per person, no duplicates). Recipes from strategist. Outputs from Clay via Cloudflare Worker webhook. segment_ids[] on leads encode client-segment membership.

clients

From Tally onboarding form. Client name, ICP criteria, target persona, pain points, USPs, case studies, DNC list, CRM platform, notification prefs. Must be approved before downstream use.

ICP, persona, pain_points live HERE — not on segments.

~30 rows

segments

Simple registry table. client_id, segment_name, segment_tag (display label), description.

segment_id (PK) is what lives on leads.segment_ids[]. segment_tag is a human-readable label (e.g., "speedsize-100k-1m-qualified").

~90 rows
UNIQUE: (client_id, segment_name). Leads reference segments by segment_id (integer), not by tag string.

leads

ONE row per person. No duplicates.

email (UNIQUE), first_name, last_name, company_name, company_domain, job_title, industry, monthly_visits, AOV, LCP, TTI, email_verified, email_verified_at.

segment_ids INT[] — array of segment_id values from segments table. e.g., ARRAY[2, 7] (where 2 = speedsize-100k-1m, 7 = solara-marketplace)

info_tags TEXT[] — free-form contextual metadata. Source, qualification, data markers. e.g., ARRAY['src:apollo', 'qualified:feb26', 'src:builtwith']

250K-500K rows
UNIQUE on email. Two separate fields: segment_ids = strict segment membership (integer references to segments.segment_id), info_tags = free-form context (source, qualification, batch markers). No FK on arrays — validated at application level.

recipes

ONE recipe per client-segment. Contains ALL approaches + selector logic + enrichment config + Clay instructions. Versioned: any change = new row, old row inactive.

approach_content: Full recipe playbook (structure TBD — Clay table recipe will be redesigned).

clay_instructions: Plain-text step-by-step for Clay operator.

~200-400 rows
Only ONE active recipe per client-segment. Naming: date-client-segment-version (e.g., 2026-03-28-hector-retail-v3).

email_outputs

Simplified for V1. One row per lead per recipe run.

lead_id, client_id, segment_id, recipe_id, recipe_version, selected_approach, company_summary, batch_id, created_at.

Subject lines: subject_line_1, subject_line_2
Email 1: email_1_variant_a, email_1_variant_b, email_1_variant_c
Email 2: email_2_variant_a, email_2_variant_b, email_2_variant_c
Email 3: email_3_variant_a, email_3_variant_b, email_3_variant_c

Variants that don't exist for a campaign stay NULL — only populated variants get pushed to sequencer.

500K-1M+ rows
selected_approach tracks which approach ran. Multiple subject lines + email variants enable A/B/C testing in the sequencer. If a recipe only produces variant A, B and C stay NULL.

What's NOT in V1

No lifecycle tracking — no last_contacted_at, cooldown_until, contacted_count. We'll figure this out later.
No campaigns table — sequencer mapping handled manually.
No sequencer API sync — no automated send/reply/bounce tracking.
No reply storage in DB — performance tracked via CSV custom fields in sequencer.
No research_report storage — research is consumed in Clay and discarded.
No DNC tags — DNC handling is a later concern.

V1 focuses on: recipes, email generation, reuse via recipe version matching, and clean handoff to campaign operator.
2. Segmentation — segment_ids INT[] + info_tags TEXT[]

Two separate fields on leads. segment_ids = strict segment membership (integer references to segments.segment_id). info_tags = free-form contextual metadata (source, qualification, batch markers).

segment_ids INT[] (Segment Membership)

segments table defines the IDs:
  segment_id=1, client=Speedsize, tag=speedsize-1m-plus-qualified
  segment_id=2, client=Speedsize, tag=speedsize-100k-1m-qualified
  segment_id=7, client=Solara, tag=solara-marketplace

leads.segment_ids uses those integers:
  john@acme.com: segment_ids = [2]
  jane@bigco.com: segment_ids = [1, 7]

Query: Speedsize 100K-1M leads
  WHERE 2 = ANY(segment_ids)

STRICT: Only segment_id values from segments table. No FK on arrays — validated at application level. GIN indexed.

info_tags TEXT[] (Additional Context)

Free-form, prefixed for organization:
  john@acme.com: info_tags = ['src:apollo', 'qualified:feb26']
  jane@bigco.com: info_tags = ['src:builtwith', 'hubspot-lead']

Prefix convention:
  src:        → data source (apollo, builtwith, salesnav)
  qualified: → when/how qualified (feb26, manual)
  batch:     → Clay batch reference

FREE-FORM: Any value allowed. For context, not for routing.

Why Two Separate Fields

segment_ids drives the system — recipes link to segments by segment_id, queries use WHERE segment_id = ANY(segment_ids), email_outputs reference client+segment. Integers, not strings — faster queries, no typos.

info_tags captures context — where the lead came from, when they were qualified, which data source enriched them. Useful for filtering and analysis but doesn't drive the pipeline.

No duplicate contacts. One person = one row. Enrichment data shared. segment_ids array handles multi-client membership. Dedup is automatic (UNIQUE on email).

4. Three Roles — Clear Boundaries

Strategist (Mayank)

Creates recipes. Owns strategy. Reviews performance.

• Pull client context from Neon
• Create recipes using Claude Code
• All approaches, selector logic, data requirements, Clay instructions — all inside one recipe
• Test on sample leads (GPT-4o-mini pipeline)
• Review emails in chat, iterate, approve
• Save recipe to Neon (versioned)
• Review performance via sequencer dashboards + CSV custom fields
• Decide: keep, kill, or rework

Does NOT touch Clay or sequencers

Clay Operator

Manages leads + TAM. Runs Clay. Hands off to campaign op.

• Sources new leads, expands TAM, works with data team
• Checks Neon: which leads are available for a client-segment?
• Pulls leads WITH existing email outputs from Neon
• Pushes ALL leads to Clay via MCP
• Sets up Clay table per recipe's clay_instructions
• Clay processes: verify + check recipe match + generate if needed
• Webhook pushes results back to Neon
• Hands clean batch (CSV) to campaign operator
• Archives Clay table

Does NOT make recipe decisions or touch sequencers

Campaign Operator

Makes campaigns live. Monitors delivery. Handles infra.

• Receives ready batch (CSV) from Clay operator
• CSV includes custom fields: segment_id, recipe_id, recipe_version, selected_approach
• Uploads to sequencer (Smartlead / Instantly / etc.)
• Makes campaign live
• Monitors reply rates, domain health, inbox placement
• Bad infra? Same emails, fresh domains (pull from Neon)

Delivery only. No Clay, no recipes, no DB writes.

6. What Goes TO Clay vs Comes BACK FROM Clay

ALL leads go to Clay. Existing email outputs travel WITH the leads so Clay can decide: pass through or regenerate.

Neon → Clay (Claude Code → Clay Webhook)

Lead data (always):
lead_id, email, first_name, last_name, company_name, company_domain, job_title, industry, segment_id, segment_name

Pre-filled enrichment (saves Clay credits):
AOV, LCP, TTI, monthly_visits, email_verified_at

Existing email outputs (for reuse check):
existing_subject_line_1, existing_email_1_variant_a, last_recipe_version, last_selected_approach

Recipe context:
client_id, current_recipe_id, current_recipe_version, info_tags

Claude Code queries Neon → POSTs one lead at a time to Clay webhook URL. One JSON object per request. Fresh leads have NULLs in existing output columns.

Clay → Neon (3 HTTP API Columns → Neon /sql)

HTTP Column 1 — Verification (ALL leads):
UPDATE leads: email_verified, email_verified_at, is_catchall

HTTP Column 2 — Enrichment (ALL leads):
UPDATE leads: LCP, TTI, AOV, monthly_visits, city, country + extra_data JSONB merge for new data points

HTTP Column 3 — Email Outputs (REGENERATE only):
INSERT email_outputs: subject_line_1/2, email_1/2/3_variant_a/b/c, selected_approach, company_summary, batch_id

Each HTTP column POSTs directly to Neon SQL-over-HTTP (/sql endpoint). No middleware. COALESCE + NULLIF pattern handles blanks. extra_data JSONB enables new data points without schema changes.
7. Clay Decision Matrix — 4 Cases

Every lead gets re-verified. Every lead's last recipe version is compared to the current active version. These two checks produce four outcomes.

Recipe Version MATCHES
Recipe Version DOESN'T MATCH (or NEW lead)
Email VERIFIED
PASS-THROUGH

Email is good. Recipe hasn't changed. Use existing emails as-is.

Pipeline does NOT run. Zero Clay credits spent on email generation. Lead passes through with existing outputs.
REGENERATE

Email is good but recipe changed (or brand new lead). Run full pipeline: enrich → select approach → generate email.

New email_output row created in Neon (with selected_approach). Old row preserved for history.
Email NOT VERIFIED
HOLD

Recipe is current but email went bad. Can't send. Lead excluded from batch.

Leads table updated: email_verified = false. Lead excluded until re-verified.
DEAD (for now)

Recipe changed AND email is bad. Don't waste pipeline credits on an undeliverable address.

Leads table updated: email_verified = false. No email generation. Lead parked.

9. Campaign Scenarios — What Happens in Each Case

A: New Client, First Campaign

All leads are new. All go through Clay. Full pipeline for every lead (REGENERATE). selected_approach tracked per lead. Hands batch to campaign op.

~5% of days

B: Same Recipe, More Leads

Mix of fresh + previously processed leads. All go to Clay. Fresh leads = REGENERATE. Previously processed with current recipe = PASS_THRU (just re-verified). Combined output handed to campaign op.

~60% of days

C: Recipe Changed

Version bumped. All leads go to Clay. Recipe match check fails for everyone (no one has outputs for new version). All leads = REGENERATE with new approach assignments. Old email_output rows preserved.

~20% of days

D: Bad Infrastructure

0% reply rate = bad sending domains. Pull same email outputs from Neon directly (no Clay needed since emails are fine). New sequencer campaign with fresh infrastructure. Zero regeneration.

~10% of days

E: A/B Test Approaches

Same recipe but random selector assigns different approaches. After 2 weeks, compare reply rates by selected_approach via sequencer custom fields. Winner gets weighted higher in selector.

As needed

You've sent 1,000 leads with v2. You change the CTA. Here's exactly what happens.

Strategist
→

2. Already Sent

1,000 leads with v2 emails: NOT touched. Live campaigns stay. Their email_output rows keep recipe_version = 2, selected_approach intact.

No action needed
→

3. Next Batch

All leads go to Clay. Recipe match check: no one has v3 outputs. Everyone = REGENERATE. Clay runs full pipeline with v3. New selected_approach assigned.

Clay Operator
→

4. Compare Later

v2 and v3 outputs both in Neon. Compare reply rates via sequencer custom fields: v2 had 2.4%, v3 has 3.2%. The change worked.

Strategist reviews
11. Cross-Client Leads — segment_ids[], Not Duplicate Rows

Same Person, Multiple Clients

leads table (ONE ROW):
id=1234 | john@acme.com | segment_ids=[3, 7]
→ where 3 = hector-retail (segment_id=3), 7 = owner-restaurant (segment_id=7)
→ Shared enrichment: AOV=$85, LCP=2.3s, monthly_visits=45K

email_outputs (SEPARATE ROWS per client-segment):
lead=1234 | client=Hector | segment_id=3 | recipe v3 | approach="ROI Calculator"
lead=1234 | client=Owner  | segment_id=7 | recipe v2 | approach="Menu Optimizer"

How it works:
Clay operator pulls leads WHERE 3 = ANY(segment_ids)
→ Gets john@acme.com along with his Hector-specific email_outputs
→ Owner.com email_outputs are ignored (different client-segment)

One person, one row, shared enrichment. Different emails per client-segment via email_outputs.
