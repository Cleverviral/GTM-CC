The Problem

We manage cold email campaigns across multiple clients. Each client has multiple segments, multiple approaches, and multiple batches. Here’s what’s breaking:

Clay Table Chaos

Every batch creates a new Clay table. Multiple clients × multiple segments × recurring batches = dozens of Clay tables with no single source of truth. Finding a specific lead's generated email means hunting through table after table.

No Email Reuse

If the approach hasn't changed and the lead was already processed, there’s no easy way to pull existing emails. The Clay operator has to manually find the right Clay table from a previous batch. Manual effort that doesn’t scale.

Version Tracking Needs Work

We track versions via the sample email repo, but it’s manual. When an approach changes (new CTA, new angle), there's no automatic record linking which version was used for which batch. Can't easily compare v2 vs v3 performance.

No Shared Enrichment Data

Same person appears across multiple client campaigns. Each time they go through Clay, enrichment is re-done from scratch. Website data, LCP scores, company info — none of it is shared or reused across clients.

How Clay Talks to Neon — 3 HTTP Columns

Each Clay table has three HTTP API enrichment columns that POST directly to Neon’s SQL-over-HTTP endpoint. Static auth. Runs per row automatically.

HTTP Column 1
Lead Verification

Runs for: ALL leads

Updates:
• email_verified
• email_verified_at = NOW()
• is_catchall

Pattern: COALESCE + NULLIF — blanks don't overwrite existing values

Why This Works

Clay’s HTTP API enrichment columns can POST to any URL. Neon exposes a SQL-over-HTTP endpoint at /sql. Authentication is a static connection string in the header — never expires, unlike JWT tokens. The SQL uses COALESCE(NULLIF($1, ''), existing_value) to handle blanks gracefully — same pattern as the Supabase upsert_contact function.

Clay Decision Matrix — 4 Outcomes

Every lead gets re-verified. Every lead’s last recipe version is compared to the current version. Two checks, four outcomes.

Recipe Version MATCHES
Recipe DOESN’T MATCH (or NEW)
Email VERIFIED
PASS-THROUGH

Existing emails reused. Zero Clay credits on generation. Just re-verified.
REGENERATE

Full pipeline runs. New email_output row created. selected_approach tracked.
Email NOT VERIFIED
HOLD

Can’t send. Excluded from batch. Lead marked unverified.
DEAD (for now)

Bad email + stale recipe. Don’t waste credits. Lead parked.

PASS-THROUGH is the money-saver. In a steady-state campaign (same recipe, recurring batches), 60-80% of leads pass through with zero generation cost. The DB pays for itself in saved Clay credits.

Campaign Scenarios — What Happens

The system handles every scenario the team encounters day-to-day.

A: New Client, First Campaign

All leads are new. All go through full pipeline (REGENERATE). Every lead gets a new email_output row with selected_approach.

~5% of days

B: Same Recipe, More Leads

Mix of fresh + existing. Fresh = REGENERATE. Existing with current recipe = PASS_THRU (zero generation cost). Biggest cost saver.

~60% of days

C: Recipe Changed

New version created. Recipe match fails for everyone. All = REGENERATE with new approach. Old outputs preserved for comparison.

~20% of days

D: Bad Infrastructure

0% reply = bad domains. Pull same emails directly from Neon (no Clay needed). New sequencer campaign, fresh infrastructure. Zero cost.

~10% of days

E: A/B Testing

Same recipe, random selector assigns different approaches. selected_approach travels as a custom field to the sequencer. Reply rate comparison happens in the sequencer dashboard — DB tracks what was sent, sequencer tracks results.

As needed

Anticipated Questions

“Why not just keep using Supabase?”

We already have the clients table built on Neon — it’s populated from Tally onboarding. Supabase works for contact storage, but it doesn’t have recipes, email_outputs, or version tracking. We’d need to build all of that on top. Neon gives us the same PostgreSQL + SQL-over-HTTP, and the migration pattern (COALESCE, domain normalization) carries over 1:1.

“Does the Clay operator need to learn SQL?”

No. The HTTP column configs are copy-paste templates. The Clay operator sets up the table like they always do — the only new thing is 3 HTTP columns instead of 1, and they point to a different URL. Claude Code handles all the Neon queries.

“What about leads that belong to multiple clients?”

One person = one row in leads. The segment_ids array holds all their segment memberships (across clients). email_outputs has separate rows per client-segment. Shared enrichment, separate emails.

“What if we need a new data field from Clay?”

Known fields (LCP, AOV, etc.) have dedicated columns. Any new or unexpected field goes into extra_data (JSONB). Clay builds a JSON string, HTTP Column 2 merges it in. Zero schema migration. Zero downtime.

“What happens to old Clay tables?”

They become disposable. Once the 3 HTTP columns push results to Neon, the Clay table can be archived or deleted. Neon has everything. Need the same emails again? Pull from DB directly.

“How does performance tracking work in V1?”

CSV exports to the sequencer include custom fields: segment_id, recipe_id, recipe_version, selected_approach. The sequencer dashboard can filter/group by these. No automated sync yet — that’s V2.

“Can I pull emails for just one client’s segment?”

Yes. email_outputs has client_id and segment_id columns. Query: SELECT * FROM email_outputs WHERE client_id = X AND segment_id = Y. You can filter further by recipe_version, batch_id, or selected_approach. Each client-segment combination is fully isolated in the output table.
