> 2. **Clay Operator** — pulls leads, pushes to Clay, manages enrichment

## System Overview

GTM-CC is CleverViral's cold email campaign system. Everything runs on:
- **Neon PostgreSQL (the "TAM + Recipe DB")** — single source of truth for leads, segments, recipes, email outputs. Connection string in `.env` as `NEON_TEST_CONNECTION_STRING`.
- **5 operational tables**: clients, segments, leads, recipes, email_outputs
- **3 roles**: Copy Strategist (admin), Clay Operator, Campaign Operator
- **Clay integration**: Webhook push (Neon → Clay) + HTTP columns push-back (Clay → Neon)

### Architecture Flow
```
Lead Sources (Apollo, LinkedIn, CSV imports)
    ↓ /add-leads (CSV import with dedup)
Neon DB — TAM + Recipe DB (leads table)
    ↓ /push-to-clay (webhook)
Clay Table (enrichment + verification + email generation)
    ↓ HTTP columns push-back to Neon
Neon DB (leads.extra_data enrichment + email_outputs)
    ↓ /export-csv
Email Sequencer (Smartlead / Instantly)
```

### Path B architecture — clients sync
The `clients` table in the TAM + Recipe DB is a **subscriber** of the `clients` table in the CleverViral Prod Neon database. Any change (new client, updated ICP, etc.) in CleverViral Prod automatically replicates to TAM + Recipe DB. The Copy Strategist never edits `clients` directly in TAM + Recipe DB — they manage clients in the CleverViral Prod app and changes flow in.

### Why this architecture
- Replaces an expensive SaaS stack with near-zero marginal cost
- Full control over prompts and pipeline
- Single orchestration tool (Claude Code in Cursor)
- n8n was explored and dropped in favor of this approach

### History — origin of the lead data
Most leads in the TAM + Recipe DB were migrated from a legacy Supabase `contacts` database in April 2026. That's why you'll see `info_tags` values like `ss-qualified-feb26`, `cleverviral-csuite-30plus-employees`, `madison-class-a`, `builtwith-data`, etc. — these are legacy qualification tags preserved from Supabase. You'll also see cross-client personalization leftovers in `leads.extra_data` (e.g. `speedsize_email1`, `solara_email_1`, `cleverviral-first-line`, `securis_email1`, `property_address` / `buyer_*`) — historical email copy generated by Supabase-era campaigns. Supabase is deprecated; all new leads come in via `/add-leads` or Clay push-back.

## Database Access

All database operations go through the `scripts/db.py` utility. Use it like this:

```python
import sys; sys.path.insert(0, 'scripts')
from db import query, read_query, write_query
```

- `read_query(sql, params=[])` — SELECT queries. Always use this.
- `write_query(sql, params=[])` — INSERT/UPDATE. Shows preview + asks confirmation.
- NEVER construct raw HTTP calls to Neon. ALWAYS use db.py.

### Available convenience functions in db.py:
- `get_clients(active_only=True)` — list clients
- `get_segments(client_id)` — list segments for a client
- `get_lead_count(segment_id)` — count leads in a segment
- `get_leads_for_segment(segment_id, limit, filters)` — pull leads with optional filters
- `get_leads_with_outputs(segment_id, recipe_id, limit)` — leads + their latest email outputs
- `get_active_recipe(segment_id)` — get the active recipe for a segment
- `get_batches(segment_id, limit)` — recent batch summaries
- `export_to_csv(rows, filepath, columns)` — write rows to CSV
- `build_clay_payload(lead, client, segment, recipe, batch_id)` — build webhook payload (core fields + extra_data flattened)
- `generate_batch_id(client_name, segment_tag)` — generate batch ID
- `push_to_clay_webhook(webhook_url, payload)` — POST payload to Clay

## Safety Rules (ALL ROLES)

1. **NEVER** run DELETE, DROP, TRUNCATE, or ALTER statements
2. **NEVER** run UPDATE without a WHERE clause
3. **NEVER** expose connection strings, API keys, or credentials in chat
4. **ALWAYS** show the query and row count before any write operation
5. **ALWAYS** confirm with the operator before executing writes
6. **ALWAYS** use parameterized queries (pass values as params, not string concatenation)
7. **ALWAYS** include LIMIT on SELECT queries (max 500 rows)
8. When reading `.env`, confirm the key exists but NEVER print the actual value

## Clay Integration

### Webhook Push (Neon → Clay): 35 core fields + flattened extra_data
When pushing leads to Clay, send every core field below PLUS every key from `leads.extra_data` flattened to the top level (each extra_data key becomes its own Clay column). See `/push-to-clay` for the exact payload builder.

| Group | Count | Fields |
|-------|-------|--------|
| Lead Identity | 7 | lead_id, email, first_name, last_name, full_name, job_title, linkedin_profile_url |
| Company | 5 | company_name, company_domain, company_website, industry, employee_count |
| Enrichment (dedicated) | 1 | monthly_visits |
| Verification | 5 | email_verified, email_verified_at, is_catchall, mx_provider, has_email_security_gateway |
| Client | 2 | client_id, client_name |
| Segment | 5 | segment_id, segment_name, segment_tag, lead_list_context, value_prop |
| Batch + Recipe | 3 | batch_id, recipe_id, current_recipe_version |
| Existing Outputs | 7 | last_recipe_version, existing_email_1/2/3_variant_a, existing_email_1/2/3_variant_b |
| extra_data flattened | N | Every `leads.extra_data` key becomes a top-level Clay column (e.g. `crux_lcp_p75`, `aov`, `product_category`, any historical personalizations) |

### HTTP Push-back (Clay → Neon)
Three HTTP columns push data back from Clay to Neon:

- **HTTP Col 1: Verification** → UPDATE leads (email_verified, is_catchall, mx_provider)
- **HTTP Col 2: Enrichment** → UPDATE leads (dedicated columns where applicable + `extra_data` JSONB merge)
  - Uses `COALESCE + NULLIF + jsonb_strip_nulls` pattern to safely merge without overwriting existing data
  - `lcp`/`tti`/`aov` are NOT dedicated columns — they go into `extra_data`
- **HTTP Col 3: Email outputs** → INSERT email_outputs (9 params: lead_id, client_id, segment_id, recipe_id, recipe_version, selected_approach, email_1/2/3_variant_a)

Use `/generate-http-query` to generate the exact JSON body for each HTTP column.

### Approach Selection
Approaches live in saved Clay templates, NOT in the webhook payload. The recipe stores the `clay_template_name` + `clay_template_link` so the operator knows which template to use.

## Clay AI Pipeline (4 Agents in Clay Table)

These agents run inside Clay as "Use AI" or "Claygent" columns. Their prompts are stored in `prompts/`.

### Agent 1: Website Scrape Agent
- **File:** `prompts/website-scrape-agent-prompt.md`
- **Model:** Claygent (web access)
- **Runs:** Every row (always)
- **Purpose:** Visit company website, extract 50-100 word summary of what they sell, who they serve, positioning
- **Output:** Text → stored in Clay column `companyProductsServices`

### Agent 2: Researcher Agent
- **File:** `prompts/researcher-agent-prompt.md`
- **Model:** Claygent (web research)
- **Runs:** Only when Research Required = YES (per approach)
- **Purpose:** 4-6 targeted web searches for approach-specific signals. Follows the RESEARCH REQUIRED section in the approach.
- **Output:** JSON `{"researchReport": "..."}` or `{"researchReport": "RESEARCH INCONCLUSIVE"}`
- **Key rule:** Does NOT repeat website scrape work — that's already in `companyProductsServices`

### Agent 3: Copywriter Agent
- **File:** `prompts/copywriter-agent-prompt.md`
- **Model:** GPT-4o-mini (Use AI column)
- **Runs:** Every row
- **Purpose:** Write Email 1 + Email 2 per prospect following the selected approach
- **Key rules:**
  - 75-85 words for Email 1
  - CTA must start with "Worth" — non-negotiable
  - Start with `{firstName},` — no "Hey" or "Hi"
  - `[blank line]` markers in approaches = line break instructions, NOT literal output text
  - MODE A (research available) vs MODE B (no research) handling
- **Output format:** `---EMAIL 1---` / `---END EMAIL 1---` / `---EMAIL 2---` / `---END EMAIL 2---`

### Agent 4: Style, Tone & Spam Checker Agent
- **File:** `prompts/style-tone-spam-checker-prompt.md`
- **Model:** GPT-4o-mini (Use AI column)
- **Runs:** Every row (post-copywriter)
- **Purpose:** 4 jobs in one pass: remove spam words (Email 1 only), humanize, casual tone check, break linear context flow
- **Key rules:**
  - Never change "Worth" CTA
  - Never add subject lines
  - Never increase word count
  - Preserve `[blank line]` markers exactly where they are
- **Output format:** `---CLEAN EMAIL 1---` / `---CLEAN EMAIL 2---`

### Model Choices
- **Claude Sonnet** — quality-critical steps (copywriting)
- **GPT-4o-mini** — cheaper steps (website summarization, research, style checking)
- **Never use** DeepSeek, Gemini, or other providers

## Campaign File Structure

Each campaign folder in `campaigns/` follows this pattern:
- `value_prop.md` — value proposition for this client-segment
- `leadlist_context.md` — who these leads are and why they're being targeted
- `data_variables.md` — what data points Clay needs to collect
- `clay_formulas.md` — Clay column formulas for enrichment
- `approach_*.md` — email approach playbooks (strategy, pattern, verbatage, examples)
- `waterfall_simulation.md` — (optional) enrichment waterfall simulation

Each approach file contains:
- STRATEGY + CORE PRINCIPLES
- PATTERN (paragraph-by-paragraph GOAL/SIGNAL/CONSTRAINT)
- VERBATAGE (approved language)
- EMAIL 2 MODE instructions
- OUTPUT EXAMPLES
- RESEARCH REQUIRED section (what the researcher agent should look for)

## Key Design Decisions

1. **value_prop on segments, not clients/recipes** — same client can target different segments with different value propositions
2. **No subject lines** — removed from the system. Subject lines are handled in the sequencer.
3. **variant_a + variant_b only** — no variant_c on email_outputs
4. **Core webhook payload + extra_data flattened** — ~35 core fields + every `leads.extra_data` key as its own Clay column. No fixed total — depends on what's in `extra_data` for a lead.
5. **Approaches in Clay templates, not webhook** — `clay_template_name` + `clay_template_link` on recipes point to saved Clay templates. Some recipes use the sentinel `"Refer to sample email repo"` for Notion-based static approaches.
6. **extra_data JSONB** — flexible catch-all for any Clay enrichment data. `lcp`, `tti`, `aov` all live here (not dedicated columns).
7. **segment_ids is an array** — a lead can be in multiple segments across clients. Same email used by multiple clients shares ONE lead row with a union of segments.
8. **info_tags vs segment_ids vs clay_table_names** — `segment_ids` drive campaigns; `clay_table_names` records Clay-table lineage (which Clay table(s) this lead has been pushed from); `info_tags` is the catch-all for everything else (legacy qualification tags like `ss-qualified-feb26`, `builtwith-data`, etc.). Three different semantic buckets, all on the same lead row.
9. **clients is subscriber-replicated** — `clients` in TAM + Recipe DB auto-syncs from CleverViral Prod. Never write to clients directly from TAM + Recipe DB.
10. **`personalizations jsonb` on email_outputs** — structured per-lead personalization data (first lines, research bullets) used to craft each email.
11. **[blank line] in approaches = formatting instruction** — the copywriter agent renders these as line breaks, NEVER as literal "[blank line]" text
12. **CSV output for sequencer** — generated emails are exported as CSV for upload to Smartlead/Instantly
13. **Reference docs as PDF** — shareable reference docs are always PDF; code-level docs (CLAUDE.md, commands, rules) stay markdown
